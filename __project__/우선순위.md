# WeLive 도메인 설계 우선순위 (Schema 기반)

## 목적

- `prisma/schema.prisma`의 관계/제약을 기준으로, 실제 구현 순서를 정해 재작업 비용을 줄이는 것
- 선행 도메인이 안정된 뒤 후행 도메인을 붙여 FK/권한/상태 충돌을 최소화

---

## 0단계: 공통 기반 (가장 먼저)

### 포함 범위

- Prisma 연결, 에러 핸들러, 인증 미들웨어, 공통 응답/페이지네이션
- enum/상태값 처리 유틸 (`Role`, `ApprovalStatus`, `ComplaintStatus` 등)

### 이유

- 모든 도메인이 공통 미들웨어/오류정책/타입을 공유함

### 완료 기준

- 빌드/테스트 파이프라인에서 공통 계층 통과
- 기본 health 체크와 에러 포맷 일관화

---

## 1단계: 계정/인증 도메인 (`User`, `AuthSession`, `Auth`)

### 포함 모델

- `User`, `AuthSession`

### 이유

- 대부분 도메인(`Notice`, `Complaint`, `Poll`, `Comment`, `Notification`, `ApartmentSchedule`)이 `authorId`/`receiverId`/`createdById`로 `User`를 참조
- 세션(`AuthSession`) 없이는 권한 제어와 실사용 테스트가 어려움

### 완료 기준

- 회원가입/로그인/토큰재발급/로그아웃
- 승인 상태(`PENDING/APPROVED/REJECTED`) 반영
- 세션 폐기(`revokedAt`) 동작 검증

---

## 2단계: 아파트/게시판 기초 도메인 (`Apartment`, `Board`)

### 포함 모델

- `Apartment`, `Board`

### 이유

- 콘텐츠 도메인(`Notice`, `Complaint`, `Poll`)이 공통으로 `apartmentId`를 요구
- `Board`는 민원/투표의 운영 단위라 선행 생성이 필요

### 완료 기준

- 아파트 생성/조회/상태관리(`ApartmentStatus`)
- 기본 게시판(`NOTICE`, `POLL`, `COMPLAINT`) 초기화 전략 확정
